## Does the modern shallow moat lead to a maintenance burden we haven't perceived, or have we reached the "free puppy" stage of AI?

I keep having a lot of discussions about AI usage and the workflows around it. I'm more fascinated by the way it's having a socio-technical impact with engineering teams in conjunction with product and design than I am at the raw speed and throughput of authoring code.

### The Shift in the Bottleneck
Previously, there was a bottleneck where, often, writing code took a while. It wasn't always the most difficult part, but it just had a tedium behind it, and meant that it took a bit. So, there were other natural processes that didn't need to be as nimble or quick or react quite as well.

Now that code can be trivialized in many ways and get us to a proof of concept or other working set of bits rather quickly, you start to see other parts in the SDLC that really haven't matured at the same rate.

### Re-evaluating Code Review
The first is code review. If you have a lot of code coming in, how do you review it and understand it? What's the value of a code review? If it's a bunch of code and you can validate that it works through testing, is a code review really that necessary? 

Well, in some cases it is because code reviews are part of a compliance process or a way to show separation of concerns or that one bad actor can't get code into production without some other compensating mechanism. And another is CodeReview is a teaching and learning experience. It's a way to collaborate and converge on styles or architectural techniques and things like that.

### Prototyping at the "Idea Stage"
But now the code often lands during the thought process and prototyping stage, or even before prototyping and just the idea stage. That really begins the discussion of how it should land in work vs. being 80% through the process or more.

And if code is coming through at this new or higher velocity, how do we maintain this all? Do we need to maintain it all, or do we just throw it at the AI and say that it should maintain it all? Are we going to have net new innovation, or is it all just cobbling together existing library tools and frameworks to make things happen? Is that enough? It might be enough.

### The Responsibility of the "Free Puppy"
So basically now we're getting code at an unprecedented rate. We have to work with it and figure it out. In some ways, it feels like the maintenance burden might be a free puppy—it looks really fun but then you realize there's a responsibility that comes with it that you're not quite sure how to deal with yet.

I'd say I have some profound outcomes here in what we should do, but I don't yet. I'm still learning, and trying to learn as much as possible as quickly as possible.
